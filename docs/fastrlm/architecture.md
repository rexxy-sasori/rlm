# FastRLM Architecture

## System Overview

FastRLM is a distributed inference architecture designed to optimize RLM (Recursive Language Model) workloads through AST-aware request routing.

## Data Flow

### 1. Request Initiation

```python
# RLM initiates a sub-call
response = rlm.completion(prompt, root_prompt=question)
```

### 2. Code Execution & AST Analysis

```python
# LocalREPL executes code and analyzes AST
class LocalREPL:
    def execute_code(self, code: str):
        # Execute the code
        result = self._execute(code)
        
        # Perform AST analysis
        ast_analysis = self._analyze_ast(code)
        
        # Call sub-LLM with metadata
        return self._call_sub_llm(code, ast_analysis, result)
```

### 3. Request with Metadata

```json
{
  "model": "qwen32b",
  "messages": [
    {"role": "system", "content": "You are a coding assistant."},
    {"role": "user", "content": "Help with this code:\nimport numpy as np\n..."}
  ],
  "extra_body": {
    "ast_analysis": {
      "complexity_score": 0.75,
      "cache_key": "a3f7b2d8e9c1f5b2",
      "estimated_tokens": 250,
      "dependencies": ["numpy", "pandas"],
      "has_loops": true,
      "has_functions": true,
      "tree_structure": "Module(body=[Import(...), FunctionDef(...)])"
    },
    "execution_context": {
      "variables": {"df": "DataFrame", "x": 42},
      "stdout": "...",
      "stderr": null
    },
    "routing_hints": {
      "priority": "high",
      "iteration_depth": 1,
      "session_id": "session_001"
    }
  }
}
```

### 4. Gateway Routing Decision

```rust
// Gateway extracts metadata and makes routing decision
fn route_request(req: Request) -> Worker {
    let ast_analysis = req.extra_body.get("ast_analysis");
    let routing_hints = req.extra_body.get("routing_hints");
    
    // Use cache key for consistent hashing
    let cache_key = ast_analysis.get("cache_key").as_str();
    let worker = consistent_hash_ring.get(cache_key);
    
    // Use complexity for batching
    let complexity = ast_analysis.get("complexity_score").as_f64();
    batcher.add_to_batch(req, complexity);
    
    // Use priority for queue ordering
    let priority = routing_hints.get("priority").as_str();
    queue.set_priority(req, priority);
    
    worker
}
```

### 5. Worker Processing

```python
# SGLang worker receives batched requests
# Uses radix attention with cache_key
# Processes and returns responses
```

## Component Responsibilities

| Component | Language | Key Responsibility |
|-----------|----------|-------------------|
| **Root LLM** | Python (via API) | High-level reasoning, problem decomposition |
| **LocalREPL** | Python | Code execution, AST analysis, metadata generation |
| **AST Analyzer** | Python | Complexity scoring, cache key generation, dependency extraction |
| **Gateway** | Rust | Request routing, batching, load balancing, worker management |
| **WASM Module** | Rust (minimal) | Optional: header validation, logging |
| **SGLang Workers** | Python/C++ | Model inference, KV cache management |

## Key Design Decisions

### 1. AST Analysis in Python

**Rationale**: Python's `ast` module is mature, well-documented, and provides complete AST manipulation capabilities. No need to reimplement in Rust.

```python
import ast

def analyze_code(code: str) -> dict:
    tree = ast.parse(code)
    return {
        "complexity": calculate_complexity(tree),
        "cache_key": generate_cache_key(tree),
        "dependencies": extract_imports(tree)
    }
```

### 2. Metadata in extra_body

**Rationale**: OpenAI-compatible API allows arbitrary fields in `extra_body`. Passes through standard clients without modification.

```python
response = client.chat.completions.create(
    model="qwen32b",
    messages=messages,
    extra_body={"ast_analysis": analysis}
)
```

### 3. Gateway in Rust

**Rationale**: SGLang Model Gateway is production-ready with built-in features:
- Request queuing and batching
- Circuit breakers and retries
- Kubernetes service discovery
- Observability (Prometheus, OpenTelemetry)

### 4. Kubernetes Pod Discovery

**Rationale**: Gateway watches Pods via K8s API, not Services. This enables:
- Direct pod IP routing
- Per-pod health checks
- Consistent hashing across pods
- Automatic scaling support

### 5. Optional WASM

**Rationale**: WASM middleware is optional. The gateway can use `extra_body` directly for routing decisions. WASM only needed for:
- Header validation
- Request transformation
- Custom logging

## Performance Optimizations

### Cache-Aware Routing

```
Request 1: cache_key="abc123" → Worker 2
Request 2: cache_key="abc123" → Worker 2 (same cache!)
Request 3: cache_key="def456" → Worker 1
```

### Smart Batching

```
Batch 1 (complexity 0.8-1.0): [Req1, Req2] → Worker 1
Batch 2 (complexity 0.4-0.6): [Req3, Req4, Req5] → Worker 2
```

### Priority Queue

```
Queue: [High: Req1, Req2] [Normal: Req3, Req4, Req5, ...]
```

## Failure Handling

### Circuit Breaker

```
Worker 1: Healthy → Unhealthy (3 failures) → Open Circuit
                                    ↓
Worker 2: Receives requests instead
```

### Retry with Backoff

```
Request → Worker 1 (fail) → Retry Worker 2 (success)
     ↓
  (exponential backoff)
```
